# 0. 목적과 범위

Futures AI는 Ver 1.1의 L3(Intelligence Layer) 컴포넌트 3-2다.
책임은 하나: **Horizon별 시장 방향의 확률과 불확실성을 출력하는 것.**

- 출력만 한다. 주문 크기·리스크·실행은 L4/L5의 몫이다.
- 포함: 내부 파이프라인, 바(bar) 생성, Triple Barrier 레이블, Horizon Expert 모델, Meta-Labeling, 불확실성, Regime 연동, 학습·재학습 설계, 모듈 구조
- 제외: 개별 Feature 정의(Ver 1.4~1.5), 옵션(Ver 1.3)

> 비유: Futures AI는 병원의 "진단과"다. 진단(방향 확률)만 정확히 내리고,
> 처방 용량(포지션 크기)과 수술(주문 집행)은 다른 과가 맡는다.
> 진단과가 처방까지 하려 들면 사고가 난다.

# 1. 내부 파이프라인 전체 구조

```
feat.{1m,3m,5m,10m,15m,30m}   intel.regime
        │                          │
        ▼                          ▼

├─────────────────────────────────────────────┤
```

# 2. 바(Bar) 생성 — 시간 바의 함정

시간 바(1분봉)는 새벽처럼 한산할 때도, 지표 발표 직후처럼 폭주할 때도 같은 1분이다.
정보량이 들쭉날쭉한 바로 학습하면 모델이 혼란스러워한다.

> 비유: 책을 "10분마다 한 장"이 아니라 "내용 한 단락마다 한 장" 넘겨야 이해가 된다.

## 2.1 이중 바 체계

| 용도 | 바 종류 | 정의 |
|---|---|---|
| 운용 기준(호환성) | 시간 바 | 1/3/5/10/15/30분 — Horizon 정의, UI 표시, 만기 시각 정렬용 |
| 학습·신호 품질 개선 | **거래대금 바(Dollar Bar)** | 누적 거래대금이 임계 T에 도달할 때마다 바 생성 |

- 1단계 구현은 시간 바로 시작한다 (단순함 우선). 거래대금 바는 Phase 2 후반 A/B 비교로 도입 결정
- 거래대금 바 임계 T: 최근 20일 일평균 거래대금 / 목표 바 개수(예: 하루 200개)로 자동 보정

## 2.2 완성봉 규율 (Bar-Close Discipline) — 운영 대원칙

**각 Horizon은 자기 완성봉 기준으로만 데이터를 확정하고 판단한다.**

- 1m 전문가: 1분봉 **완성 시점**의 데이터로, 매 1분마다 판단
- 30m 전문가: 30분봉 **완성 시점**의 데이터로, 매 30분마다 판단
- 진행 중인 미완성 봉의 값(변하는 고가·저가·종가)은 어떤 전문가도 사용하지 않는다

이유 3가지:

1. **재현성(no repainting)**: 미완성 봉은 계속 바뀐다. 완성봉만 쓰면 학습 데이터와 실시간 추론이 정확히 같은 것을 본다 — 백테스트·실시간 parity(Ver 1.1 §2-3)가 구조적으로 보장된다
2. **전문가 독립성의 시간축 완성**: "전문가는 서로를 모른다"(1장) + "각자 자기 시계로만 움직인다". 30m 전문가를 1분마다 깨우지 않는다
3. **부하 절감**: 30m 전문가는 하루 13회만 추론 → Hot Path 경량화

운영 규칙:

| 항목 | 규칙 |
|---|---|
| Feature 계산 | 틱으로 증분 누적하되, **발행(`feat.{h}`)은 해당 Horizon 완성봉 확정 시점에만** |
| 봉 확정 판정 | 거래소 시각 기준 경계(예: 09:30:00.000) + 지연 틱 유예 500ms 후 확정 |
| 의견 유효기간 | 각 전문가 의견에 `valid_until = 다음 완성봉 시각` 포함 — Aggregator가 신선도 반영 |
| 레이블 정렬 | Triple Barrier 진입 시점도 완성봉 시점 — 학습·추론 시점 동일 |
| 예외 | Kill Switch·Risk 감시는 봉과 무관하게 틱 단위 상시 가동 (안전장치는 시계를 기다리지 않는다) |

# 3. 레이블링 — Triple Barrier 상세 설계

## 3.1 구조

진입 시점마다 3개의 벽을 세운다. **먼저 닿는 벽이 레이블이다.**

```
가격
 ▲  ── 상단 배리어 (+k_up × ATR) ──── 닿으면 레이블 +1
 │      ╱╲    ╱╲
 │     ╱  ╲__╱  ╲___          시간 배리어 (H 경과)
 │    ╱              ╲        ── 닿으면 레이블 0 (수익부호로 세분화 가능)
 │ ── 하단 배리어 (−k_dn × ATR) ──── 닿으면 레이블 −1
 └──────────────────────────────▶ 시간
```

## 3.2 Horizon별 배리어 파라미터 (초기값 — 전부 Walk-Forward로 재추정)

| Horizon | 시간 배리어 H | 상/하단 폭 (×ATR(주기)) | 비고 |
|---|---|---|---|
| 1m | 3분 | 0.5 | 비용 대비 폭이 작아 가장 엄격한 비용 검증 필요 |
| 3m | 9분 | 0.8 | |
| 5m | 15분 | 1.0 | 기준 Horizon |
| 10m | 30분 | 1.2 | |
| 15m | 45분 | 1.5 | |
| 30m | 90분 | 2.0 | Regime 영향 최대 |

- 폭은 ATR(변동성) 배수로 정의 → 조용한 장·거친 장에서 자동으로 벽 간격이 조절된다
- **비용 반영 레이블**: 상·하단 도달이 왕복 비용(수수료+슬리피지 추정)을 넘지 못하면 0으로 강등 — "이겼지만 남는 게 없는 거래"를 이기는 것으로 배우지 못하게 한다

## 3.3 샘플 가중치 (Uniqueness)

겹치는 레이블(동시에 열려 있는 여러 진입)은 정보가 중복된다.
각 샘플의 가중치 = 평균 고유도(average uniqueness)로 부여하고, 부스팅 학습의 `sample_weight`로 전달한다.

# 4. Horizon Expert 모델 설계

## 4.1 전문가 1개의 사양 (6개 공통 틀)

| 항목 | 사양 |
|---|---|
| 모델 | LightGBM 분류기 (3-class: +1/0/−1) — CPU 최적, Ver 1.0.1 §4.1 방침 |
| 앙상블 | 동일 구조 × 시드/배깅 5개 = 미니 앙상블 (분산 → 불확실성 원료) |
| 입력 | 해당 Horizon Feature 집합 (Ver 1.5에서 확정, 실효 20~50개) |
| 출력 | `P(+1), P(0), P(−1)` + 앙상블 표준편차 |
| 클래스 불균형 | class weight로 보정 (0 레이블이 다수가 되는 게 정상) |
| 제약 | 단조 제약(monotone constraints)을 명백한 Feature에 적용 가능 (예: 체결강도↑ → P(long) 비감소) — 과적합 방지 |

## 4.2 Horizon별 특성 차이 (설계 관점)

| Horizon | 성격 | 주 Feature 계열 (Ver 1.5 상세) | 특이 사항 |
|---|---|---|---|
| 1m | 마이크로구조 | OFI, 호가 불균형, 체결강도, Micro Price | 비용이 승부처 — Meta-Labeler 임계 최고 |
| 3m | 마이크로+모멘텀 | 위 + 단기 모멘텀, Volume Delta | |
| 5m | 단기 추세 | VWAP 괴리, ATR, 모멘텀 | 기준 전문가 (가장 먼저 구현) |
| 10m | 추세 지속성 | 추세 강도, 수급 흐름 | |
| 15m | 수급 | 외국인/기관/프로그램 누적 | |
| 30m | 국면 | Regime, Basis, IV, 시장폭, 매크로 | Regime AI와 입력 일부 공유 |

**구현 순서: 5m → 15m → 30m → 10m → 3m → 1m.**
(5m이 비용·데이터 양·신호 품질의 균형점. 1m은 비용 검증이 가장 어려우므로 마지막.)

# 5. Meta-Labeling — 2단 판단 구조

## 5.1 왜 2단인가

1차 모델(방향)은 "어느 쪽인가"에는 강하지만 "지금이 배팅할 때인가"에는 약하다.
역할을 분리하면 각자 단순해지고, 승률 필터가 독립적으로 학습된다.

> 비유: 사수와 관측수다. 사수(1차)는 조준만, 관측수(2차)는 "바람이 나쁘다, 쏘지 마라"만 판단한다.
> 한 사람이 둘 다 하면 둘 다 어설퍼진다.

## 5.2 사양

- 학습 데이터: 1차 모델이 낸 신호들만 모아, "그 신호가 실제로 맞았는가(이익이었는가)"를 이진 레이블로
- 입력: 1차 확률·마진, 앙상블 분산, Regime, 실현변동성, 스프레드, 이벤트 근접도(D-day), 시간대(장 초반/중반/후반)
- 출력: `P(신호가 옳음)` — 임계 미달 시 해당 신호 **기각(0으로 강제)**
- 모델: 얕은 LightGBM (깊이 3~4) — 필터는 단순해야 필터답다
- Horizon마다 별도 Meta-Labeler를 둔다 (1m의 "나쁜 조건"과 30m의 "나쁜 조건"은 다르다)

# 6. 불확실성 정량화

두 겹으로 산출한다.

1. **앙상블 분산**: 미니 앙상블 5개의 P(+1) 표준편차 → 즉시 계산, Hot Path 사용
2. **Conformal Prediction**: 최근 교정 구간(예: 20일)의 예측-실현 오차 분포로 신뢰구간 산출 → "P(long)=0.72 ± 0.09" 형태

활용 규칙:

- `uncertainty` 필드로 `intel.futures`에 항상 포함 (Ver 1.1 스키마의 `uncertainty`)
- 불확실성 > 임계 → Aggregator가 해당 전문가 의견의 가중치를 자동 축소
- L4 Position Sizer가 불확실성 페널티로 재사용 (Ver 1.0.1 §1.5)

# 7. Regime 연동 — 가중치 매트릭스

Regime AI의 국면 출력에 따라 Aggregator의 전문가 가중치와 Meta-Labeler 임계값이 바뀐다.

## 7.1 초기 가중치 매트릭스 (Walk-Forward로 재추정하는 출발값)

| Regime \ Horizon | 1m | 3m | 5m | 10m | 15m | 30m | Meta 임계 보정 |
|---|---|---|---|---|---|---|---|
| 추세 상승 | 0.5 | 0.8 | 1.0 | 1.2 | 1.2 | 1.5 | 기본 |
| 추세 하락 | 0.5 | 0.8 | 1.0 | 1.2 | 1.2 | 1.5 | 기본 |
| 횡보 | 1.2 | 1.0 | 1.0 | 0.8 | 0.6 | 0.4 | +0.05 (더 엄격) |
| 고변동성 | 0.8 | 0.8 | 1.0 | 0.8 | 0.8 | 0.8 | +0.10 |
| 이벤트 근접 | 0.3 | 0.3 | 0.5 | 0.5 | 0.5 | 0.5 | +0.15 (사실상 관망) |
| UNKNOWN | 0.5 | 0.5 | 0.5 | 0.5 | 0.5 | 0.5 | +0.10 (보수 모드) |

- 논리: 추세장은 긴 Horizon 우대, 횡보장은 짧은 Horizon 우대, 이벤트 전엔 전부 감쇠
- 이 표 자체가 학습 대상이다 — 분기마다 Walk-Forward 성과로 재추정

## 7.2 Aggregator 출력

```
통합 방향 점수 S = Σ_h [ w(regime, h) × (P_h(+1) − P_h(−1)) × meta_h × (1 − u_h) × f_h ]
  meta_h: Meta-Labeler 통과 여부(0/1),  u_h: 정규화 불확실성
  f_h:    신선도(freshness) — 완성봉 직후 1.0, valid_until 도달 시 0으로 선형 감쇠
```

- 완성봉 규율(§2.2) 때문에 각 의견의 나이가 다르다. 29분 전의 30m 의견은 곧 갱신될 낡은 의견이므로 f_h로 감쇠하고, 갱신 직후엔 온전히 반영한다

- `intel.futures`에는 **Horizon별 원시 의견 전부 + 통합 점수 S**를 함께 발행 (Meta Decision Engine이 재량으로 사용)
- XAI: 통합 점수에 대한 기여 상위 5개 (전문가×Feature)를 함께 발행 — UI 근거 표시의 원천

# 8. 학습 데이터와 검증 설계

## 8.1 데이터 구성

- 최소 확보 목표: **틱/호가 2년치** (Regime 다양성 확보 — 상승·하락·횡보·급변 모두 포함되어야 함)
- 학습 표본 단위: 바 기준 (3장), 레이블은 Triple Barrier (3장)
- 이벤트 일(지표 발표·만기일)은 제외하지 않는다 — 대신 Feature로 명시(이벤트 근접도)하여 모델이 구분하게 한다

## 8.2 Walk-Forward 스킴 (Ver 1.0.1 §1.2 구체화)

```
|── 학습 6개월 ──|─ 검증 1개월 ─|            → 창을 1개월씩 전진, 반복
      ↑ Purged K-Fold(5) + Embargo(1일)로 내부 튜닝
```

- Purge: 검증 구간과 레이블이 겹치는 학습 샘플 제거 / Embargo: 검증 직전 1일 학습 제외
- 하이퍼파라미터 탐색은 학습 창 내부에서만 — 검증 창은 단 1회만 사용
- 최종 지표: 전체 검증 창 연결 성과의 **Deflated Sharpe** (시도 횟수 보정)

## 8.3 Registry 통과 기준 (초기값)

| 지표 | 기준 |
|---|---|
| 비용 차감 후 Sharpe (검증 연결) | > 1.0 |
| Deflated Sharpe p-value | < 0.05 |
| 최대낙폭 | < 학습기 자본 기준 한도 |
| 검증 창별 성과 일관성 | 음(−) 성과 창 비율 < 40% |
| 1m 전문가 추가 기준 | 평균 슬리피지 2배 가정에서도 양(+) 기대수익 |

미달 시 Registry 등록 거부 (Ver 1.1 §6-2 Validator 소관).

## 8.4 재학습 주기

| 대상 | 주기 | 방식 |
|---|---|---|
| Horizon Expert | 월 1회 정기 | 최신 창으로 재학습 → candidate → shadow 20거래일 → 승격 심사 |
| Meta-Labeler | 월 1회 (Expert와 동시) | Expert 신호 재수집 후 재학습 |
| Conformal 교정 구간 | 매일 | 최근 20일 롤링 (재학습 아님, 교정만) |
| Regime 가중치 매트릭스 | 분기 1회 | Walk-Forward 재추정 |
| 긴급 재학습 | 수시 | Feature 중요도 붕괴 or 성과 급락 감지 시 (Self Evaluation 트리거) |

**즉시 반영은 없다.** 어떤 재학습 결과도 shadow를 거치지 않고 live가 되지 않는다 (Ver 1.1 §6-4).
