# 1. 최종 통합 아키텍처

```
════════════════════════ 1개 PC = Full Stack Instance ═══════════════════════
                                                     (N대 복제 → 독자 거래)
  L1 DATA        Collector → Normalizer → Archiver(Parquet) + Event Calendar
                        │ (Message Bus: Redis — 모든 통신의 유일 통로)
  L2 FEATURE     Feature Store(v2026.xx) → Realtime Engine
                        │ 완성봉 규율: 각 Horizon 확정 시점에만 발행
  L3 INTELLIGENCE
     ┌ Regime AI (HMM+규칙) ──────────────┐
     ├ Futures AI: Expert×6 → Meta-Labeler → Aggregator ┤→ Meta Decision
     └ Options AI: Vol Engine → 매트릭스 → Evaluator ────┘   Engine
                        │ decision.intent (XAI 근거 포함)
  L4 CAPITAL     Cost Model → Risk Engine(거부권) → Position Sizer
                        │ capital.order_request        [Kill Switch 상주]
  L5 EXECUTION   Execution Engine → Broker Adapter ⇄ 거래소
                        │              (↕ 동일 인터페이스: Digital Twin)
  L6 LEARNING    Trainer → Validator → Registry → Shadow → 승격 (야간)
  ────────────────────────────────────────────────────────────────────────
  OBS            구조화 로깅 · Command Center UI · 알림(INFO/WARN/CRITICAL)
═════════════════════════════════════════════════════════════════════════════
```

5대 불변 원칙 (전 문서의 결정 사항 요약):

1. **완성봉 규율** — 각 Horizon은 자기 완성봉 확정 시점에만 판단. 안전장치만 틱 단위 (Ver 1.2 §2.2)
2. **거부권 구조** — 수익은 L3가 만들고 생존은 L4가 지킨다. Risk 거부는 항소 불가 (Ver 1.1)
3. **동일 인터페이스** — 백테스트·페이퍼·실전은 설정 한 줄 차이 (Digital Twin, Ver 1.0.1 §2.1)
4. **Shadow 없이 live 없다** — 모델·피처셋 승격은 예외 없이 20거래일 검증 후 (Ver 1.6 §9.2)
5. **복제 배포** — 릴리스(코드+모델 번들) 하나로 N대 PC가 독자 거래. 차이는 instance.yaml뿐 (Ver 1.1 §7)

구체 예시로 전 레이어를 관통한다. — 10:35:00, 5분봉 완성 시점.

```
10:35:00.000  5m 완성봉 확정 (유예 500ms 후) → feat.5m 발행
10:35:00.520  [L3] 5m Expert 앙상블: P(+1)=0.71±0.04 (교정 후)
              Meta-Labeler: 스프레드 정상·이벤트 없음 → 통과 (P=0.68 > τ=0.60)
10:35:00.525  [L3] Aggregator: 15m(신선도 0.8)·30m(0.6)도 양(+) 동조
              → 통합점수 S=+0.42, 불확실성 0.11 (Conformal)
              Regime="추세상승" → 긴 Horizon 가중 적용
10:35:00.530  [L3] Meta Decision: Options AI는 NO_OPTION (IV Rank 45, 우위 없음)
              → 의도 = LONG (미니선물), 확신도 0.71, 근거 top5 첨부
10:35:00.535  [L4] Cost Model: 왕복 예상비용 0.8틱 → Net ER = +2.1틱 > 0 ✓
              Risk Engine: 증거금 사용률 31%<40% ✓, 일일손실 −0.4%>−2% ✓ → 승인
              Sizer: Vol Target × 1/4 Kelly × (1−0.11) → 미니 3계약
10:35:00.540  [L5] Execution: 지정가 분할(2+1), TTL 30초
10:35:03      체결 완료. exec.fill → UI 갱신, 로그에 Feature 스냅샷+모델버전 결합 저장
   ⋮
10:50:00      Triple Barrier 시간배리어(15분) 도달 전 익절선 터치 → 청산
              Holding Policy A유형: 보유 15분, 당일 무포 원칙 유지
장 마감 후     이 거래는 Self Evaluation 집계에 편입 → 내일의 시스템을 조금 바꾼다
```

이 흐름 전체가 로그로 재현 가능하다 — "왜 샀는가"에 시스템은 언제나 답할 수 있다.

# 3. Meta Decision Engine — 확정 사양

## 3.1 결정 규칙 (우선순위 순)

```
① sys.kill 활성 → 무조건 NO TRADE
② Regime="이벤트" 또는 UNKNOWN → NO TRADE (신규 기준)
③ 전문가 의견 분산 > 0.25 → NO TRADE (의견이 갈리면 배팅하지 않는다)
④ |S| < 0.20 → NO TRADE (우위 부족)
⑤ S ≥ +0.20 → LONG 후보 / S ≤ −0.20 → SHORT 후보
⑥ Options AI 후보의 Net ER가 선물 Net ER × 1.3 초과 시 → OPTION 우선
   (옵션은 비용·복잡도가 크므로 30% 이상 우위일 때만 선택)
⑦ 방향 후보 + 옵션 후보 병존 && 상관 노출 중복 → L4가 합산 노출로 판정
```

## 3.2 출력 계약

- `decision.intent`: 의도, 확신도(교정 확률), 불확실성, 근거 top5, 모델버전, 지연 추적 (Ver 1.1 §4.3 스키마)
- **NO TRADE도 근거와 함께 발행한다** — 침묵이 아니라 판단이다. UI에 "왜 쉬는가"가 표시된다

## 7.1 회의체 (Governance) — 아무것도 조용히 묻히지 않게 하는 장치

데이터를 모으고 계측을 심어도, **결정하는 자리가 없으면 항목은 조용히 사라진다.**
(미륵이 실증: 311차에 관찰된 이상 패턴이 근본원인 미규명 상태로 방치 → 335차에야 버그로 확정)

### 회의 체계

| 회의 | 시점 | 소요 | 안건 (자동 생성) | 산출물 |
|---|---|---|---|---|
| **일간회의** | 매 거래일 장마감 후 | 15~30분 | 당일 손익·거래 리뷰, 이상 이벤트(WARN+), 개발중 디버그 결과, "라이브 미검증" 항목의 당일 검증 결과 | DECISION_LOG 기록, 개선항목 티켓화, 익일 계획 |
| **주간회의** | 금요일 장마감 후 | 1~2시간 | 주간 성과 분해(Horizon·전략·시간대별), Shadow vs 챔피언 중간 성적, 검증캠페인 판정, **미결항목 에이징 전수 점검** | 주간 판정 기록, 보류 항목 기한 재설정 |
| **월간회의** | 월말 주간회의 확장 | +1시간 | 재학습 번들 승격 심사(사람 승인, Ver 1.6 §9.2), 릴리스 발행 결정 | release-YYYY.MM 승인 |
| **분기회의** | 분기말 | 반나절 | 피처셋 재선정 결과 승인, Regime 가중치 재추정, Discovery 방어전 판정, KPI 대비 리뷰 | feature_set 버전, 정책 개정안 |

### 운영 규칙 3가지 (묻힘 방지의 핵심)

1. **안건은 시스템이 만든다** — 회의 전 자동 생성되는 안건 리포트:
   "라이브 미검증" 태그 목록, 기한 초과 미결항목(aging), 데이터는 쌓였는데 분석·판정이 안 된 계측(섀도 게이트·counterfactual) 목록, 신규 WARN/CRITICAL 집계.
   사람이 안건을 기억할 필요가 없게 한다 — 기억은 시스템이, 판단은 사람이.
2. **모든 안건은 결정으로만 종료된다** — 허용되는 결론은 3가지뿐:
   **채택**(즉시 티켓화) / **보류**(반드시 기한 명시, 기한 도래 시 자동 재상정) / **폐기**(사유 기록).
   "검토하기로 함"으로 끝나는 안건은 존재할 수 없다 — 그것이 묻힘의 시작이다.
3. **에이징 규칙**: 미결항목이 30일 초과 시 주간회의 최상단 강제 배치, 60일 초과 시 "폐기 또는 즉시 착수" 양자택일 강제.

### 1인 개발 체제에서의 회의

혼자여도 회의는 연다 — 형식이 규율을 만든다. 자동 생성된 안건 리포트를 놓고
정해진 시간에 결정을 기록하는 행위 자체가 회의다. AI 어시스턴트와의 리뷰 세션
(미륵이의 딥다이브 세션 방식)을 일간·주간회의의 실무 형태로 계승한다.

# 8. 실전 진입 — 4단계 승격 절차 (관문 수치 확정)


# 12. 맺음 — 이 시스템이 믿는 것

이 설계서 전체를 관통하는 믿음은 세 가지다.

**첫째, 우위(Edge)는 예측이 아니라 규율에서 나온다.**
비용을 세고, 검증을 통과하고, 정해진 시간에 나오는 것 — 화려한 모델보다 이것이 수익을 만든다.

**둘째, 생존이 복리를 만든다.**
Kill Switch, 거부권, 보유기간 상한, 무포 오버나이트 — 이 모든 브레이크는 속도를 줄이기 위해서가 아니라
**내일도 달리기 위해** 존재한다. 파산하지 않는 자만이 복리를 경험한다.

**셋째, 진화는 리듬이다.**
매일 평가하고, 매월 도전자를 세우고, 분기마다 재료를 갈고, 이기는 것만 남긴다.
기관은 규모로 이기지만, 우리는 **반복 실험의 속도**로 이긴다.

> 시장은 내일도 열린다. 서두르지 않되, 멈추지 않는다.
