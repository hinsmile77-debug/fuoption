# 0. 총평 — 마흐디가 보여주는 것

마흐디는 개발 2.5주차의 초기 시스템이지만, **미륵이의 교훈이 이미 상당 부분 반영된 모던 스택**이다
(uv·TimescaleDB·docker-compose·pytest 341개·order_state_machine·known-value 단위테스트·절대경로 금지).
그럼에도 사고는 났다 — 이번엔 늪(상태 동기화)이 아니라 **수도관(데이터 파이프라인)과 외부 세계(브로커 API·DB·OS)의 경계**에서.

> 비유: 미륵이가 "정글의 늪 지도"를 남겼다면, 마흐디는 "상수도 공사의 함정 지도"를 그리는 중이다.
> 물(데이터)은 파이프 어디가 새는지 말해주지 않는다 — 새는 곳마다 계량기를 달아야 안다.

특기사항: 마흐디는 **옵션 관측 시스템**이라 그 교훈이 메시아 Options AI(Ver 1.3)와 Vol Engine에 직결된다.

# 1. 교훈 13선 (L16~L28) — 사건 · 근본 원인 · 메시아 반영

## L16. theta 오버플로우 — 스키마 자릿수가 데이터를 조용히 삼킨 사건 (최대 교훈)

- **사건**: `theta NUMERIC(8,6)` 컬럼(정수부 최대 2자리)에 KIS가 주는 **원화 단위 theta(-1,722 ~ -9,461)** 삽입 → 상시 오버플로우. 만기 임박 위클리일수록 theta가 커져 **하루 내내 단조 누적 유실**(분당 30행→10행), 장 마감 무렵엔 근접 두 만기가 전량 소실 — **감마 리스크가 가장 큰 구간이 정확히 비는** 최악의 형태. 5일간(07-16 "원인불명") 정체를 몰랐고, 원시값 로깅 추가 후에야 확정.
- **근본 원인**: ① 외부 API 필드의 **실측 범위·단위를 모른 채** 스키마를 설계 (이론적 Greeks 단위 가정 vs 브로커의 원화 환산값), ② 삽입 실패가 행 단위로 조용히 스킵.
- **메시아 반영**:
  1. **외부 필드 실측 우선 원칙**: 브로커 API의 모든 수치 필드는 최소 5거래일 원시값 수집 → 실측 범위표 작성 후 스키마 확정 (마흐디의 `KIS_RAW_FIELD_RANGES.md` 방식을 표준 절차화)
  2. Greeks·가격 필드는 자릿수 인색한 NUMERIC 대신 **여유 있는 타입** + Feature 스키마 `range_check`(Ver 1.4 §1.1)로 이상치를 계측으로 잡는다 — 저장은 관대하게, 검증은 엄격하게
  3. **단위(unit)는 스키마 필수 항목** — "theta"가 아니라 "theta(원/일)"인지 "theta(pt/일)"인지 명기 (Ver 1.4 스키마 `unit` 필드가 이 사고의 백신)
  4. 삽입 실패율이 임계 초과 시 WARN — 행 단위 스킵의 누적을 침묵시키지 않는다

## L17. 마이그레이션 라이브 미적용 — 파일의 스키마 ≠ DB의 스키마

- **사건**: 마이그레이션 SQL 2건이 커밋만 되고 실행 중인 컨테이너엔 미적용(docker-entrypoint-initdb.d는 볼륨 최초 생성 시 1회만 실행) → 기동 직후부터 INSERT 전부 `UndefinedColumn` 실패 → rollback 누락 잠복 버그와 결합해 연쇄 실패 → **COCKPIT이 23분간 합성(가짜) 데이터 표시**.
- **메시아 반영**:
  1. **기동 시퀀스에 마이그레이션 자동 적용 + 스키마 정합성 검증** 포함 — 코드가 기대하는 스키마 버전과 DB 실제 버전 불일치 시 기동 거부 (Ver 1.1 §7.3 자가 점검에 항목 추가)
  2. DB 예외 처리에 rollback 의무화 — 커넥션 오염 연쇄 차단
  3. 멀티 PC 복제 배포 시 마이그레이션은 릴리스 설치 스크립트의 일부 (신규 PC든 기존 PC든 동일 경로)

## L18. 폴백이 가짜 데이터를 조용히 보여주는 위험

- **사건**: 대시보드가 DB 조회 실패 시 "합성 리플레이"로 폴백 — 개발엔 유용했지만, 장중 사고 시 **사용자가 실시간인 줄 알고 가짜 데이터를 봄** (125회 폴백을 나중에야 인지).
- **메시아 반영**: **폴백은 시끄럽게** — Command Center에 데이터 소스 상태 배지(LIVE / DELAYED / FALLBACK / SYNTHETIC)를 상시 표시하고, FALLBACK 전환 시 WARN 알림. 운영 모드에서 합성 데이터 폴백은 아예 금지 (빈 화면이 가짜 화면보다 낫다).

## L19. 거래소 상품 체계는 문서가 아니라 실측으로 확정

- **사건**: 위클리 옵션이 N/O(월요일 만기)·L/M(목요일 만기) **별개 상품**임을 이틀에 걸쳐 실측으로 발견 (첫날은 "L/M 없음"으로 오판 — 그 주에 우연히 N/O만 존재). 종목 마스터파일의 실제 컬럼 순서가 공식 참고 스크립트와 다름. 만기 판별은 월물구분코드가 아니라 한글종목명 정규식이 정답.
- **메시아 반영**: Broker Capability Matrix(미륵이 L9)에 **상품 체계 실측 검증** 차원 추가 — 위클리 월/목 2북 구조, 코드풀, 만기 판별 방법을 실측 교차검증 후 상수화하고 근거(실측 일자·방법)를 주석으로 남긴다. 메시아 Ver 1.3의 위클리 취급(§8)은 이 2북 구조를 전제로 설계 유지.

## L20. 레이트리밋 — 정규장 절반의 데이터를 잃고 배운 것

- **사건**: REST 폴링 4개 루프가 각자 호출해 TPS 초과 → 정규장 절반 데이터 유실. 모의투자 리밋도 문서와 달라 실측 재조정(2건/초→1건/초). "작업 후 sleep" 방식 폴링은 작업 지연이 누적돼 5분 간격에 4분씩 유실.
- **메시아 반영**:
  1. **공유 레이트리미터 단일 인스턴스** — 모든 REST 호출이 하나의 리미터를 통과 (주문 경로 단일화와 같은 철학: 예산은 한 곳에서 관리)
  2. 적응형 백오프 (실패 시 즉시 감속, 연속 성공 시 서서히 복귀)
  3. **폴링은 절대시각 고정 틱 스케줄링** — "작업 후 sleep" 금지 (완성봉 규율과 정합: 시계에 맞춰 움직인다)

## L21. 타임존 — naive datetime이 두 번 사고 낸다

- **사건**: ① naive-KST 타임스탬프가 TIMESTAMPTZ 컬럼에 UTC로 오라벨링(데이터 자체가 시간 거짓말), ② naive/aware 비교 TypeError로 헬스체크 **전체 다운** (정규장 첫 실측에서 발견).
- **메시아 반영**: 이미 원칙(UTC+거래소시각 병기, Ver 1.1)이 있으나 규율로 강화 — **naive datetime 생성 전면 금지** (lint 규칙으로 강제: `datetime.now()` 금지 → `now_utc()` 유틸만 허용). DB 저장은 TIMESTAMPTZ + UTC 통일, 표시만 KST 변환.

## L22. 한 행의 삽입 실패가 루프 전체를 죽인다

- **사건**: 위클리 도입 직후 옵션 레그 1건의 DB 삽입 실패가 관측 루프 전체를 중단시킴.
- **메시아 반영**: 이미 설계에 있음 (Ver 1.1 §1-2 Normalizer 격리 큐) — 강조 확장: **수집 루프의 모든 항목 처리는 개별 격리** (한 종목·한 행의 실패는 그 항목만 격리 큐로, 루프는 계속). 실패 시 원본 페이로드 로깅 (L16의 5일 미궁이 원시값 로깅 부재 탓이었다).

## L23. 프로세스 수명주기 — 창 제목으로 프로세스를 관리하지 마라

- **사건**: 사고 수습 중 수동 재시작한 프로세스가 배치의 창 제목 규약(`start "Mahdi ..."`)을 안 거침 → 장후 자동 종료 스크립트(taskkill WINDOWTITLE)가 **아무것도 못 찾고 그냥 종료**. 게다가 기록("관측 루프 무중단 유지")과 실측(OS 생성시각 08:40 재시작)이 모순 — 수동 개입은 기록조차 왜곡한다.
- **메시아 반영**:
  1. 프로세스 관리는 창 제목·프로세스명이 아니라 **PID 파일 + 자가 등록** (기동 시 인스턴스가 Redis에 자기 PID·시작시각 등록, 종료 스크립트는 그것을 본다)
  2. **종료·강제청산(Forced Flat) 시퀀스는 자기검증 필수** — 마흐디의 결정("종료 시퀀스 자기검증 없이는 배포 금지")을 메시아 전체 원칙으로 승격: 종료 스크립트는 실행 후 "정말 죽었는지" 확인하고 결과를 기록한다
  3. 수동 개입도 정식 경로로 — 수동 재시작 스크립트를 제공해 규약 우회를 원천 차단

## L24. 로그 회전 경계 — 오탐 분석의 함정

- **사건**: 회전된 로그 파일에 이전 실행분이 섞여 "오늘 500 에러 620건"이라는 오탐 분석 → 같은 날 정정 (실제 오늘은 0건).
- **메시아 반영**: 모든 로그에 **세션 경계 마커**(기동 시각·인스턴스 ID·git SHA를 첫 줄에) + 분석 도구는 기본적으로 "마지막 기동 이후" 구간만 조회. 일간회의 안건 자동 생성기(Ver 2.0 §7.1)도 세션 경계 기준으로 집계한다.

## L25. 화석 데이터 — 스키마·라벨 변경의 뒤처리

- **사건**: 위클리 분리(weekly→weekly_mon/thu) 후 구 라벨 `series='weekly'` 179건이 화면에 영구 노출 — 화이트리스트 필터 + DB 삭제로 수습.
- **메시아 반영**: 라벨·enum 변경은 3종 세트 의무 — ① 코드 변경, ② 구 데이터 마이그레이션(변환 또는 삭제), ③ 조회부 유효값 화이트리스트. 마이그레이션 파일에 이 3종 체크리스트를 주석으로 강제.

## L26. 모의투자 ≠ 실전 — 환경 차이 매트릭스

- **사건**: 옵션체인 REST 1회 조회 불가(모의), WS 모의 전용 도메인 없음, "모의 미지원" 문서의 수급 API가 실전 도메인으로 호출 성공, CBOT/CME 미승인으로 해외선물 조회 실패 → yfinance 폴백. "실거래 전환 시 재검토 필요 항목"을 별도 정리하기 시작.
- **메시아 반영**: **환경 차이 매트릭스**를 Capability Matrix의 축으로 추가 (기능 × {모의, 실전} × {구현, 실측검증}). 임시 데이터 소스(yfinance 등)는 `_source` 컬럼으로 출처를 데이터에 새기고, G3(소액 실전) 진입 관문에 "임시 소스 잔존 0건" 항목 추가 (Ver 2.0 §8 연동).

## L27. cmd.exe 배치의 함정 — 로직은 Python으로

- **사건**: 배치파일 괄호 파싱 크래시, CRLF 필수, 한글 mojibake(chcp 65001), ERRORLEVEL 시점 고정(미륵이 B66 재현) — 배치 스크립트에서만 4건.
- **메시아 반영**: **배치파일은 "Python을 부르는 한 줄"만** — 날짜 연산·조건 분기·문자열 처리는 전부 Python 스크립트로 (마흐디 07-20 결정을 표준화). 배치 공통 헤더 템플릿(chcp 65001·CRLF·로그 리다이렉션) 고정.

## L28. 운영점검 → 번호 매긴 고도화 → 순차 구현 (마흐디가 가장 잘한 것, 계승)

- **관찰**: 07-16 운영점검보고서가 고도화 아이디어에 번호(1~7번)를 매기고, 이후 세션들이 그 번호를 하나씩 구현("아이디어 4번 구현")하며 **하나도 묻히지 않고 전부 소화**. 오탐이면 같은 날 정정 기록. 매일 보고서 → 이상점 → fix → 고도화의 리듬이 이미 작동.
- **메시아 반영**: 이것이 바로 Ver 2.0 §7.1 회의체의 실증 사례 — **일일 운영점검보고서 양식(장전/장중/장후 이상점 + 번호 매긴 고도화 항목)을 메시아 일간회의의 표준 산출물로 채택**. 번호가 곧 티켓이고, 티켓은 에이징 추적된다.

# 3. 금지 계명 추가 (미륵이 10계명에 이어 11~15)

11. 외부 API 필드의 실측 범위·단위를 모른 채 스키마를 확정하지 않는다
12. 폴백·합성 데이터를 실데이터인 척 조용히 표시하지 않는다 — 폴백은 배지와 경보를 동반한다
13. naive datetime을 만들지 않는다 — 시간은 태어날 때부터 UTC-aware다
14. 종료·강제청산 시퀀스를 자기검증 없이 배포하지 않는다
15. 코드의 스키마와 DB의 스키마가 같다고 가정하지 않는다 — 기동 시마다 검증한다

# 4. 남기는 말

미륵이는 늪의 지도를, 마흐디는 수도관의 지도를 남겼다.
미륵이의 교훈이 마흐디에서 이미 작동하고 있다는 사실(모던 스택·테스트·기록 문화)은
**레슨런 반영이 실제로 효과가 있다는 증거**다. 메시아는 두 장의 지도를 모두 들고 출발한다.

> 같은 강에 두 번 빠지는 팀은 지도가 없어서가 아니라, 지도를 읽지 않아서 빠진다.
> 메시아의 회의체(Ver 2.0 §7.1)는 이 지도를 매일 읽는 자리다.
