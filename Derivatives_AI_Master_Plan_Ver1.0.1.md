# 1. Ver 1.0 검토 — 최고의 수익률을 위한 보완 사항

Ver 1.0은 "무엇을 만들 것인가"는 훌륭하게 정의했다.
그러나 실전 수익률을 결정하는 것은 화려한 예측 엔진이 아니라 **검증, 비용, 리스크의 3중 안전장치**다.

> 비유: Ver 1.0은 강력한 엔진을 가진 스포츠카 설계도다.
> 그런데 브레이크(리스크), 연비계산(거래비용), 주행시험장(백테스트)이 빠져 있다.
> 실제 레이스에서 우승하는 차는 엔진이 아니라 이 세 가지로 결정된다.

## 1.1 거래비용 · 슬리피지 모델 (최우선 보완)

Ver 1.0에는 수수료, 세금, 슬리피지, 시장충격 비용이 없다.
단기 Horizon(1~5분) 전략은 예측이 맞아도 비용 때문에 손실이 나는 경우가 대부분이다.

보완 사항:

- **Cost Model 모듈 신설**: 수수료 + 슬리피지(호가 스프레드 기반) + 시장충격(주문크기/유동성 비율)
- 모든 백테스트와 실시간 의사결정에서 **비용 차감 후 기대수익(Net Expected Return)** 으로 판단
- 판단 기준 변경: `예측 확률 > 임계값` → `Net Expected Return > 0 AND 확률 > 임계값`

## 1.2 검증 프레임워크 (Backtest / Walk-Forward) 부재

Phase 2(Feature)에서 Phase 3(모델)로 바로 넘어가면 안 된다.
**Phase 2.5 — Validation Infrastructure** 를 신설해야 한다.

- **Walk-Forward 검증**: 과거 학습 → 미래 검증을 굴려가며 반복 (시험문제를 미리 보고 시험 보는 것 방지)
- **Purged K-Fold + Embargo**: 시계열 데이터 누수(leakage) 차단
- **Deflated Sharpe Ratio**: 수백 번 실험 중 우연히 좋았던 전략 걸러내기
- 백테스트 → 페이퍼 트레이딩(모의) → 소액 실전 → 정상 운용의 **4단계 승격 절차**

## 1.3 레이블링 전략 명시

"5분 뒤 오르면 LONG"식 단순 레이블은 노이즈가 크다.

- **Triple Barrier Method**: 익절선 / 손절선 / 시간마감 3개 벽 중 먼저 닿는 것으로 레이블 생성
- **Meta-Labeling**: 1차 모델이 방향을, 2차 모델이 "이 신호를 믿을까?"를 판단 → 승률과 배팅 크기 분리

## 1.4 Feature 300~500개의 과적합 위험

Feature가 많을수록 좋은 게 아니다. 데이터 대비 Feature가 많으면 모델은 과거를 암기한다.

- Feature 채택 절차 신설: 후보 등록 → 단독 예측력 검정 → 기존 Feature와 상관 중복 제거 → Walk-Forward 생존 시 승격
- **Feature Importance 붕괴 감지**: 중요도가 급변하는 Feature는 자동 격리
- 실효 Feature 목표: Horizon당 20~50개 (전체 사전은 300~500개 유지하되, 실전 투입은 소수 정예)

## 1.5 포지션 사이징의 구체화

Risk AI에 "Position Size"가 항목으로만 존재한다. 수익률의 절반은 사이징에서 나온다.

- **Volatility Targeting**: 목표 변동성 대비 현재 변동성으로 포지션 조절 (파도가 높으면 돛을 줄인다)
- **Fractional Kelly (1/4~1/2 Kelly)**: 확률 우위에 비례한 배팅, 단 파산 방지를 위해 축소 적용
- 일일 손실 한도 / 연속 손실 시 자동 축소 규칙 명문화

## 1.6 옵션 AI의 델타 헤징 및 포지션 수명 관리

옵션은 진입보다 **보유 중 관리**가 수익을 결정한다.

- Delta Hedging 모듈: 옵션 포지션의 방향 노출을 선물로 중화하는 정책 (헤지 주기, 밴드)
- 포지션 수명주기: 진입 → 조정(Roll/Adjust) → 청산 조건을 전략별로 정의 (Iron Condor의 날개 조정 등)
- **IV Rank/Percentile 필터**: IV가 역사적으로 높을 때만 매도 전략 허용

## 1.7 Kill Switch와 이상 감지 (안전장치)

자율 AI일수록 비상 정지 장치가 필수다.

- **Hard Kill Switch**: 일일 최대손실, 주문 오류율, 데이터 지연 임계 초과 시 전 포지션 청산 + 신규 주문 차단
- **Anomaly Detector**: 체결가-호가 괴리, 피드 단절, 모델 출력 이상치 감시
- 사람의 수동 개입(Override)은 항상 최상위 권한

## 1.8 Market Regime AI의 구체화

Ver 1.0에서 Regime AI는 이름만 있다.

- Regime 정의 명시: 추세상승 / 추세하락 / 횡보 / 고변동성 / 이벤트(만기·지표발표) 등 5~7개
- HMM(Hidden Markov Model) 또는 클러스터링 기반 + 규칙 기반 하이브리드
- **Regime별로 하위 AI의 가중치와 임계값이 달라지는 구조**를 명문화

# 2. 기관 수준을 넘어서는 프레임워크 — 추가 작업 제안

기관을 이기는 방법은 기관을 흉내 내는 것이 아니다.
기관은 규모(자본·인력)로 이기고, 개인은 **속도(반복 실험 속도)와 집중(니치 시장)** 으로 이긴다.

## 2.1 Digital Twin 시뮬레이터 (최고 우선순위 제안)

모든 틱 데이터를 이벤트로 저장(Event Sourcing)하고, 과거 어느 날이든 **그날의 시장을 그대로 재생**할 수 있는 시뮬레이터를 만든다.

> 비유: 비행기 조종사가 실기 전에 시뮬레이터에서 수천 번 나는 것과 같다.
> AI를 실전에 투입하기 전, 과거 3년의 시장을 수천 번 "다시 살아보게" 한다.

- 호가창 수준 재생 + 자기 주문의 시장충격 모사
- 백테스트 코드와 실전 코드가 **동일 인터페이스**를 사용 (시뮬레이터 ↔ 실거래소 교체만으로 전환)
- 이것이 있으면 Self Evolution(Phase 5)이 안전해진다: 진화된 모델을 시뮬레이터에서 먼저 검증

## 2.2 Feature Store + Model Registry (연구 공장화)

- **Feature Store**: 모든 Feature의 정의·계산코드·버전을 중앙 관리. 학습 시점과 실시간 시점의 계산이 항상 동일하도록 보장
- **Model Registry**: 모델 버전, 학습 데이터 구간, 성과지표를 기록. 언제든 롤백 가능
- 실험 추적(MLflow 등)으로 "무엇을 시도했고 왜 버렸는지"를 자산화 → 개인이 기관의 리서치팀을 이기는 핵심

## 2.3 Shadow Trading (그림자 운용)

실전 모델과 별개로, 차기 후보 모델들이 **실시간 시장에서 가상 주문**을 내며 성적을 쌓는다.

- 현역 모델 vs 후보 모델의 실시간 성적 비교
- 후보가 일정 기간 이기면 자동 승격 심사 → Self Evolution의 실체가 이것
- 승격/강등 이력을 모두 기록 (스포츠 리그의 승강제와 같다)

## 2.4 불확실성 정량화 (Uncertainty-Aware AI)

기관 모델과의 차별화 포인트.

- **Conformal Prediction**: "LONG 확률 78%"가 아니라 "이 예측의 신뢰구간" 자체를 출력
- 불확실성이 높으면 자동으로 포지션 축소 또는 NO TRADE
- 앙상블 분산을 불확실성 지표로 사용 (전문가들의 의견이 갈리면 배팅하지 않는다)

## 2.5 이벤트 · 텍스트 인텔리전스 모듈

- 경제지표 발표, FOMC, 만기일 등 **이벤트 캘린더 AI**: 이벤트 전후 포지션 정책 자동 전환
- LLM 기반 뉴스/공시 요약 → 이벤트 리스크 점수화 (예측이 아니라 **위험 회피**용으로 사용)

## 2.6 실행 최적화 강화학습 (장기 과제)

Execution AI를 규칙 기반으로 시작하되, Digital Twin이 완성되면 강화학습으로 주문 집행(분할, 타이밍)을 학습시킨다. 시뮬레이터 없이는 착수 금지.

# 3. Trading UI — "Command Center" 설계안

기술 기반: **웹 대시보드 (FastAPI + WebSocket 백엔드, React + 차트 라이브러리 프론트)**
멀티 PC·원격 모니터링에 유리하고, 실행 PC와 화면 PC를 분리할 수 있다.

> 비유: 조종석(Cockpit) 설계와 같다. 조종사는 수백 개 계기 중
> "지금 봐야 할 것"만 크게, 나머지는 이상 시에만 눈에 띄게 만들어야 한다.

## 3.1 디자인 원칙

- **다크 테마 고정**: 배경 `#0D1117` 계열, 장시간 응시 피로 최소화
- 색은 의미로만 사용: 상승/LONG = 청록(`#00C9A7`), 하락/SHORT = 로즈(`#FF5C7A`), 경고 = 앰버, 중립 = 그레이
  (국내 관습인 빨강=상승 옵션도 설정에서 전환 가능하게)
- 숫자는 등폭 폰트(JetBrains Mono 등), 변화 시 플래시 애니메이션으로 시선 유도
- **3초 원칙**: 화면을 3초만 봐도 "포지션, 손익, AI 의견, 위험" 4가지가 파악되어야 한다

## 3.2 화면 구성 (5개 존)

```
┌───────────────────────────────────────────────────────────┐
│ ① Top Bar: 총자산 · 일일 PnL · 리스크 게이지 · KILL SWITCH │
├───────────────┬──────────────────────┬────────────────────┤
│ ② AI Decision │ ③ Market View        │ ④ Position &       │
│   Panel       │   (차트 + Regime)     │   Risk Panel       │
├───────────────┴──────────────────────┴────────────────────┤
│ ⑤ Bottom: 실행 로그 · 이벤트 캘린더 · Self-Eval 요약       │
└───────────────────────────────────────────────────────────┘
```

**① Top Bar (항상 고정)**

- 일일 PnL(실현+평가), 일일 손실한도 대비 게이지
- **KILL SWITCH 버튼**: 물리적으로 가장 오른쪽 위, 2단 확인 후 전량 청산·주문 차단
- 시스템 상태등: 데이터 피드 / 모델 / 주문 라인 각각 녹·황·적

**② AI Decision Panel (핵심 차별화)**

- 최종 결정 카드: `LONG / SHORT / OPTION / NO TRADE` + 확신도 게이지(불확실성 구간 포함)
- **의사결정 근거(XAI)**: "왜?"를 상위 5개 Feature 기여도 바로 표시
  (예: 호가불균형 +32%, 외국인 순매수 +21%, IV 하락 −15%…)
- Horizon별 전문가 의견 매트릭스: 1/3/5/10/15/30분 모델의 방향·확신도를 한 줄 히트맵으로
- 전문가들의 의견이 갈리면(분산 큼) 패널 테두리가 앰버로 변해 "지금은 불확실"을 시각화

**③ Market View**

- 메인 캔들차트 + AI 진입/청산 마커 오버레이
- **Regime 리본**: 차트 상단에 현재 국면(추세/횡보/고변동/이벤트)을 색 띠로 연속 표시
- 옵션 탭: IV Surface 3D/히트맵, Skew 곡선, 만기별 OI 분포, Greeks 히트맵
- 마이크로 탭(단타용): 호가창 잔량 불균형, 체결강도, OFI 실시간 스트림

**④ Position & Risk Panel**

- 보유 포지션 테이블: 종목, 방향, 수량, 평단, 평가손익, **남은 만기일**, 포트폴리오 Greeks 합산(Δ, Γ, Θ, V)
- 리스크 계기판: VaR, 최대낙폭 진행률, 노출도(선물/옵션/현금 배분 도넛)
- Theta Decay 시계: 옵션 보유 시 "오늘 시간가치로 녹는 금액" 실시간 카운터

**⑤ Bottom Zone**

- 실행 로그(주문→체결 지연시간 포함), 이벤트 캘린더(지표발표 D-day, 만기 D-day)
- Self-Evaluation 미니보드: 최근 200회 승률, Profit Factor, 모델별 성적 스파크라인

## 3.3 알림 체계 (3단계)

| 등급 | 예시 | 전달 방식 |
|---|---|---|
| INFO | 신호 발생, 체결 완료 | 화면 토스트 |
| WARN | 불확실성 급증, 피드 지연 | 토스트 + 사운드 |
| CRITICAL | 손실한도 접근, Kill Switch 발동 | 전면 배너 + 사운드 + 모바일 푸시(텔레그램 등) |

## 3.4 구현 스택 제안 (Python 친화)

- 백엔드: **FastAPI + WebSocket** (실시간 스트림), 데이터는 Redis pub/sub에서 구독
- 프론트: React + **Lightweight Charts**(TradingView 오픈소스, 캔들) + ECharts/Plotly(IV Surface, 히트맵)
- 1단계 프로토타입은 **Streamlit/Dash로 빠르게** 만들어 화면 구성을 검증한 뒤, 2단계에서 React로 이관하는 것을 권장
- 원격 접속: 실행 PC의 대시보드를 같은 네트워크 또는 VPN(Tailscale 등)으로 열람 — 포트를 인터넷에 직접 노출하지 않는다

# 4. 시스템 환경 및 코딩 지침 (현 PC + 멀티 PC 배포)

## 4.1 현재 환경 전제와 설계 방침

전제: **일반 PC, GPU 없음** (Windows, Python 기반)

따라서 설계 방침은 명확하다:

- 모델은 **Gradient Boosting 계열(LightGBM / XGBoost / CatBoost) 중심**으로 설계한다.
  탭형(tabular) 금융 Feature에서는 딥러닝 대비 성능이 비슷하거나 우수하고, CPU에서 빠르다.
- 딥러닝(LSTM/Transformer)은 "GPU 확보 후 실험 항목"으로 분류하고, 초기 아키텍처의 필수 요소로 두지 않는다.
- 데이터 처리는 **Polars(권장) 또는 Pandas 2.x + PyArrow**, 루프 대신 벡터화. 저장은 **Parquet** 표준.
- 실시간 경로(호가 수신→Feature→추론→주문)는 지연 예산을 정한다: 예) 총 50ms 이내, 구간별 측정 로그 필수.

> 비유: GPU 없는 지금은 "경차로 랠리에 나가는 것"이 아니라,
> 애초에 경차가 유리한 코스(부스팅 모델 + 탭형 데이터)를 고르는 전략이다.

## 4.2 멀티 PC 배포 아키텍처 — 독립 복제(Replication) 모델

멀티 PC의 의미는 역할 분산이 아니라 **완성본의 복제 배포**다.
완성된 시스템 1세트를 여러 PC에 그대로 설치하면, **각 PC가 자기 계좌·자기 자본으로 독자 거래**한다.

```
[개발/학습 PC]  ──릴리스(코드+모델 번들)──▶  [PC-01 독자 거래]
                                        ├▶  [PC-02 독자 거래]
                                        └▶  [PC-03 독자 거래]
      각 PC = Full Stack (수집→Feature→AI→리스크→실행→UI) + 자기 계좌
      PC 간 실시간 의존 없음 — 한 대가 죽어도 나머지는 계속 거래
```

- 인스턴스별 차이는 **설정 파일 하나**(계좌, 자본, 한도, 종목)뿐 — 코드·모델은 전 PC 동일 릴리스
- PC 내부 모듈 간 통신은 로컬 Message Bus(Redis)로만 한다 (직접 함수 호출 금지) → 한 PC 안에서도, 복제해도 동일하게 동작
- 메시지 스키마는 **Pydantic 모델로 단일 정의**, 전 PC가 같은 패키지 버전을 사용
- 시간은 반드시 **UTC + 거래소 타임스탬프** 병기, 전 PC NTP 동기화 (틱 정렬이 틀어지면 모든 Feature가 오염된다)
- 학습은 대표 PC 1대에서 수행 → 모델 번들을 릴리스로 각 PC에 배포 (상세: Ver 1.1 §7)

## 4.3 프로젝트 구조 표준 (Monorepo)

```
fuoption/
├─ pyproject.toml            # 의존성·버전 단일 관리
├─ configs/                  # 환경별 YAML (dev / paper / live)
│   └─ live.yaml             # 시크릿은 .env / 환경변수로 분리 (git 금지)
├─ src/fuoption/
│   ├─ core/                 # 메시지 스키마(Pydantic), 상수, 시간 유틸
│   ├─ data/                 # 수집기, 정규화, 저장
│   ├─ features/             # Feature Store (Feature = 클래스 1개 = 파일 1개)
│   ├─ models/               # 학습·추론, Model Registry
│   ├─ strategy/             # Futures / Options / Meta Decision
│   ├─ risk/                 # Cost Model, 사이징, Kill Switch
│   ├─ execution/            # 주문, 브로커 API 어댑터
│   ├─ simulator/            # Digital Twin (실행기와 동일 인터페이스)
│   └─ ui/                   # FastAPI 대시보드 백엔드
├─ tests/                    # pytest (Feature 계산 회귀 테스트 필수)
└─ scripts/                  # 배포·운영 스크립트
```

핵심 원칙: **simulator와 execution이 같은 인터페이스(추상 클래스)를 구현**한다.
전략 코드는 자신이 모의인지 실전인지 몰라야 한다.

## 4.4 코딩 지침 (Python)

1. **Python 3.11+ 고정**, 가상환경(venv 또는 uv) 필수, 의존성은 `pyproject.toml`에 버전 고정(lock)
2. **타입힌트 100% + Pydantic 검증**: 돈이 걸린 코드에서 "아마 float일 것"은 금지
3. 가격·손익 계산은 **float 대신 Decimal 또는 정수 틱 단위** 사용 (부동소수점 누적 오차 방지)
4. **설정과 코드의 분리**: 임계값·수수료율·계좌번호는 코드에 하드코딩 금지 → YAML/환경변수
5. **구조화 로깅(JSON)**: 모든 의사결정은 "입력 Feature 스냅샷 + 모델 버전 + 출력"을 함께 기록 → 사후 재현 가능
6. 실시간 경로는 **비동기(asyncio)**, 학습·백테스트는 **multiprocessing** — 둘을 한 프로세스에 섞지 않는다
7. 테스트 규율: Feature 계산 함수는 고정 입력→고정 출력 회귀 테스트 필수. 모델보다 **파이프라인 버그가 계좌를 깬다**
8. Git 규율: `main`은 항상 실행 가능 상태, 전략 변경은 브랜치 + 백테스트 리포트 첨부 후 병합
9. **Docker(또는 uv 환경 고정)로 배포**: "내 PC에선 됐는데"를 원천 차단. Windows 실행기는 초기엔 네이티브로 두고, 수집기·학습기부터 컨테이너화
10. 장애 복구: 모든 프로세스는 재시작 시 **마지막 상태(포지션·주문)를 브로커 API에서 재조회**해 복원한다. 로컬 기억을 신뢰하지 않는다
