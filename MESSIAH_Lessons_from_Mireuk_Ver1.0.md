# 0. 총평 — 미륵이가 남긴 유산

미륵이(선물 자동진입청산 시스템)는 실거래까지 도달한 실전 플랫폼이며,
그 dev_memory는 **660개의 "돈 주고 산 교훈"** 이다.
버그의 대부분은 모델이 아니라 **주문 상태 동기화, 피처-모델 정합성, 게이트 적층, 프로세스 구조**에서 터졌다.

> 비유: 미륵이는 지도 없이 정글을 개척한 1차 원정대다. 대원 대부분을 잃은 곳은
> 맹수(시장)가 아니라 **늪(상태 동기화)과 아군 오발(게이트 적층)** 이었다.
> 메시아는 1차 원정대가 피로 그린 지도를 들고 출발한다.

**단일 최대 교훈: 미륵이에게는 시뮬레이터가 없었다.**
363차 세션의 다수가 "실전에서 버그 발견 → 수정 → **라이브 미검증** 태그 → 다음 실전이 곧 테스트"의 루프였다.
메시아가 Digital Twin(Ver 2.0 로드맵 W9~11)을 **모델보다 먼저** 만드는 이유가 바로 이것이다.

# 1. 교훈 15선 — 사건 · 근본 원인 · 메시아 반영

## L1. 유령 포지션 — 주문 pending 선등록 누락 (최대 단일 손실 사건)

- **사건**: 틱 하드스톱(S0-C)이 청산주문을 pending 미등록으로 전송 → 실체결이 "외부 진입"으로 오인 → 반대방향 유령 포지션 자동 생성 → 재차 하드스톱. 하루 -675만원(당일 손실의 사실상 전부). EmergencyExit(B104)·시간청산·부분청산(B106)에서도 동종 버그 반복.
- **근본 원인**: 청산 경로가 6개+ (수동/TP/하드스톱/틱스톱/시간/비상)로 흩어져 있고, "pending 선등록 → 전송 → 체결 매칭" 패턴이 경로마다 제각각. **새 경로를 추가할 때마다 같은 race condition이 재발**.
- **메시아 반영**:
  1. 주문 전송 함수는 시스템 전체에 **단 하나** — 모든 진입·청산·헤지·비상 경로가 `OrderGateway.submit()` 하나를 통과한다. pending 등록은 이 함수 내부에서 원자적으로 수행 (우회 불가능한 구조로 봉쇄)
  2. "브로커에 나간 주문 ≠ 내부 상태"를 전제로 한 Reconciler 상시 대사 (Ver 1.1 §5-3)
  3. **체결 매칭 실패는 유령 포지션 생성이 아니라 CRITICAL 정지** — 모르는 체결이 오면 거래를 멈추고 사람을 부른다

## L2. 실전이 테스트장 — "라이브 미검증" 백로그

- **사건**: 수정의 상당수가 "라이브 미검증" 태그로 배포. 검증 스크립트가 운영 상태파일을 오염시킨 사고도 발생.
- **메시아 반영**: Digital Twin을 최우선 구축(W9~11). **모든 수정은 replay 재생 테스트 통과 후 배포** (Ver 1.1 §6 장중 배포 금지 + §7.3 자가 점검). 운영·검증 환경은 물리적으로 분리 — 검증 코드는 운영 DB·상태파일에 접근 불가.

## L3. 학습-추론 피처 공간 불일치 (조용한 성능 살인자)

- **사건**: 백필이 12피처만 저장, feature_names.pkl 오염, 모델 119 vs 120차원 불일치, SHAP 레지스트리가 배포 모델과 다른 피처셋 참조 → 대시보드 3시간+ 침묵 실패(DEBUG 레벨로 숨겨짐).
- **근본 원인**: 피처 정의가 코드 여러 곳에 분산, "레지스트리 따로 모델 따로", 불일치가 침묵으로 진행.
- **메시아 반영**: 이미 설계에 반영됨 — Feature Store 단일 정의(Ver 1.4), 번들 내 `feature_set.yaml`이 유일한 진실(Ver 1.6 §9), **버전 불일치 시 추론 거부 + WARN**(Ver 1.6 §10). 추가 강제: 불일치·차원 오류는 어떤 경우에도 DEBUG가 아닌 **ERROR 레벨** — 침묵 실패 금지.

## L4. 진입 0건의 날들 — 게이트 적층의 복리 비용

- **사건**: min_conf·CoherenceGate·MetaGate·CB③·Checklist·Layer2·Contrarian·Hurst 게이트가 수년간 겹겹이 추가 → 완전 관망일 반복(5/21·5/22·6/8 리뷰). 앙상블 conf 상한 자체가 임계값 미달(최대 57.1% vs min_conf 58%). 퇴역한 30m 정확도로 정상 신호까지 차단하는 게이트(CB③-P4)도 발견.
- **근본 원인**: ① 교정 안 된 확률로 임계값 씨름 (conf가 진짜 확률이 아님), ② 게이트 신설 시 counterfactual 계측 없이 라이브 투입, ③ 게이트 간 상호작용을 아무도 전체적으로 보지 못함.
- **메시아 반영**:
  1. **Isotonic 교정 필수**(Ver 1.6 §6.1) — 임계값이 실제 확률 의미를 갖게 한 후에만 사용
  2. **게이트 신설은 섀도 계측 → 승격** 원칙: 신규 차단 로직은 최소 20거래일 "차단했더라면" 기록 후 실차단 승격 (미륵이 말기에 도달한 결론을 메시아는 첫날부터)
  3. 차단 계층 개수 상한: Meta-Labeler + Risk Engine + Kill Switch **3계층으로 고정** (Ver 1.1) — 새 게이트는 기존 계층에 흡수, 계층 신설 금지
  4. **진입 퍼널 일일 리포트 표준 탑재**: 신호→차단 사유별 카운트를 매일 UI·리포트로 (미륵이 342차의 교훈)

## L5. 스케일러·정규화 사고 계열

- **사건**: z=+92 폭발 피처, CLIP 누락, 갭오픈 절대가격 피처 z폭발, ScalerWarmup 0-패딩, 스케일러 격리 버그, 재적합 타이밍 이슈로 세션 다수 소모.
- **메시아 반영**: **구조적 회피** — LightGBM은 스케일 불변이므로 스케일러 서브시스템 자체를 두지 않는다(Ver 1.6). 대신 Feature 스키마의 `range_check`(Ver 1.4 §1.1)로 이상치를 계측·격리. 절대가격 수준 피처 금지 — 전부 비율·괴리·Z-정규화 형태로만 정의 (Ver 1.4 사전이 이미 준수).

## L6. 레이블·임계값 표류

- **사건**: 고정 HORIZON_THRESHOLDS가 시장 레벨(1200pt) 변화로 부적합 → FLAT 과소 → CB③ 오발동(B100). rolling σ 레이블과 고정 임계의 괴리. class_weight `balanced`가 추세장 FL 100% 고착. SGD 인접 표본 29/30 중복(레이블 해머링)으로 단방향 붕괴.
- **메시아 반영**: 이미 구조적으로 회피 — Triple Barrier가 ATR 배수라 시장 레벨에 자동 적응(Ver 1.2 §3.2), uniqueness 가중치가 중복 표본 문제를 직접 처리(§3.3). 추가 규칙: **임계값과 class_weight는 항상 한 세트로 변경**하고 Walk-Forward 재검증을 동반 (미륵이 원칙 계승).

## L7. 장중 온라인 학습(SGD)의 배신

- **사건**: 장중 SGD가 콜드스타트 루프, 붕괴 후 meta_raw=0 고착, 편향 증폭 등으로 수십 세션 소모. 결국 역할 축소·전용화로 후퇴.
- **메시아 반영**: **장중 학습 전면 금지**를 명문화. 학습은 야간 배치(Ver 1.6 §7), 장중에는 추론과 Conformal "교정"만(재학습 아님, §6.2). 시장 적응은 온라인 학습이 아니라 불확실성 확대→자동 보수화로 해결.

## L8. 단일 프로세스 모놀리스의 대가

- **사건**: main.py 12,000줄+, `_ts_` 몽키패치로 클래스 메서드 광범위 오버라이드 → 죽은 코드를 고치는 헛수고, elif 순서 버그, EOD 처리 중 UI 완전 먹통, 장중 재학습이 파이프라인 블로킹(CB⑤ 수천 ms), Qt 스레드·QTimer 사고 다수. 감사보고 P0도 "main.py 책임 집중"을 최우선 지적.
- **메시아 반영**: 이미 아키텍처가 답 — 프로세스 분리 + Redis 통신(Ver 1.1 §7.1), UI는 별도 프로세스의 구독자. 추가 코딩 규율:
  1. **몽키패치 전면 금지** — 오버라이드는 정식 상속·주입으로만
  2. **파일 상한 500줄** — 초과 시 분리 리팩터링을 그 자리에서
  3. 죽은 코드는 발견 즉시 삭제 (보관은 git 이력이 한다)

## L9. 브로커 API의 함정들

- **사건**: TR 코드 오류(CpSyrNew7212→7221), 지수 조회 미지원 TR, COM STA 데드락(30초 타임아웃), 세션만료 access violation, 미니선물 만기 코드 구독으로 틱 미수신(B97), unfilled_qty 상수 0(B75), **잘못된 종목코드로 KOSDAQ150 선물 실진입(B71)**, 재시작 후 코드 불일치로 다른 종목에 청산주문(B73).
- **메시아 반영**:
  1. **Broker Capability Matrix** (감사보고 R4 채택): 기능별 "구현됨 vs 실측 검증됨"을 분리 관리 — 검증 안 된 기능은 사용 금지
  2. **종목코드 프로브 표준화**: 근월물 코드는 저장값 신뢰 금지, 기동 시마다 프로브로 확정 (미륵이 D49·D84 계승) + 주문 직전 코드-포지션 일치 게이트 (D50)
  3. 브로커 어댑터는 replay 통합 테스트 대상 — TR 헤더 매핑은 실측 로그로 검증 후 상수화 (D48 계승)

## L10. 관측 시스템 자체의 오작동 (계측의 계측)

- **사건**: 정상 체결 로그가 예외 밀도에 집계돼 Degraded Mode 29분 오발동(307차), PSI 상시 0.000(부트스트랩 부재로 죽은 계측), FQAdj 실효 0으로 수개월 무효, SYSTEM.log 하루 200MB, 태그 하나에 WARNING과 CRITICAL 변형이 섞여 exclude 불가.
- **메시아 반영**:
  1. **"지표가 장기간 불변이면 그 자체가 경보"** — 모든 감시 지표에 heartbeat 검증 (죽은 계측 자동 탐지)
  2. 로그 태그 규율: 태그 1개 = 심각도 1개. 같은 태그로 레벨 혼용 금지
  3. 헬스 집계는 정책성·진단성 로그를 애초에 별도 채널로 분리 (사후 exclude 목록 관리 금지)
  4. 로그 볼륨 예산: 일 상한 설정, 초과 시 샘플링 강등

## L11. 멀티 PC 드리프트 — 커밋 안 된 로컬 수정

- **사건**: MW0601 PC의 코드 수정이 커밋 없이 로컬에만 존재 → MW0602와 대조하며 수동 재구현(299차). 두 PC의 스케일러·모델 상태 불일치 복구 작업(358차).
- **메시아 반영**: 이미 원칙화 — **릴리스 번들로만 배포, PC별 임의 수정 금지**(Ver 1.1 §7). 추가 강제: 인스턴스 기동 자가 점검에 "git SHA + 번들 해시 = 릴리스 매니페스트 일치" 검증 포함. 불일치 PC는 거래 개시 거부.

## L12. 상태 영속화·복원의 함정

- **사건**: ui_prefs 복원 전 덮어쓰기(B68·B120), 재시작 후 SHAP 윈도우 미복원, 전일 예측이 당일 집계에 혼입(carry-over), stale 캐시 오염(B112), CB 상태 영속화 범위 논쟁.
- **메시아 반영**: 이미 원칙화 — **무상태 프로세스 + 상태는 Redis/브로커/Parquet**(Ver 1.1 §6). 복원 순서 표준: 외부 소스 조회 → 대사 → 그 후에만 구독 시작. "로컬 기억을 신뢰하지 않는다"가 모든 복원 코드의 첫 줄 주석이 된다.

## L13. UI 수동 배선의 반복 버그

- **사건**: "MireukDashboard 메서드 추가 시 DashboardAdapter 바인딩 동시 등록 필수"가 반복패턴 버그로 등록될 정도 — 위젯 참조 미저장, 갱신 함수 미호출, 파라미터 누락 등 UI 표시 버그 수십 건.
- **메시아 반영**: UI는 **구독 모델** — 백엔드가 UI 함수를 호출하는 것이 아니라, UI가 `ui.*` 토픽을 구독해 스스로 그린다(Ver 1.1 OBS). 새 지표 추가 = 토픽 발행 추가일 뿐, UI 배선 의무가 구조적으로 없다.

## L14. 확신도·정확도 지표의 자기기만

- **사건**: 화면 63% vs 실제 판정 58% 이중 기준, conf=100% FLAT(반올림 탈출), Brier 과신 패널티 상시 발동, 정확도 0.0% 표시 버그, Mid-Conf Blind Spot(60~85% 구간 침묵).
- **메시아 반영**: 확률의 진실성은 교정으로 강제(Ver 1.6 §6) + **Brier Score·신뢰도 다이어그램을 Validator 관문으로**(§8) — "말한 확률만큼 맞는가"를 배포 전에 검사. 표시값과 판정값은 같은 소스에서만 파생 (이중 기준 구조적 차단).

## L15. 기록 문화 — 미륵이가 가장 잘한 것 (계승)

- **관찰**: dev_memory의 DECISION_LOG(증상→원인→결정→Why→How to apply→검증) 형식, 차수 세션 체계, 정기점검 딥다이브, 트레이드 리뷰, counterfactual 섀도 계측, 진입 퍼널 리포트 — 이 문화가 660개 교훈을 자산으로 만들었다.
- **메시아 반영**: **dev_memory 체계를 첫 커밋부터 도입** — 동일 형식의 DECISION_LOG.md / SESSION_LOG.md / CURRENT_STATE.md / NEXT_TODO.md + trade_review/. 추가 개선: "라이브 미검증" 태그에 **검증 기한**을 붙이고 NEXT_TODO에서 자동 추적 (미검증 백로그 방치 방지).
- **회의체로 완성**: 기록이 자산이 되려면 **결정하는 자리**가 필요하다 — 일간회의(장마감 후)·주간회의(금요일)·월간 승격 심사·분기 리뷰의 정기 회의체를 운영 규정으로 확립. 안건은 시스템이 자동 생성(미검증 목록·에이징 초과·미판정 계측), 모든 안건은 채택/기한부 보류/사유부 폐기로만 종료 — "조용히 묻힘"의 구조적 차단. 상세: Ver 2.0 §7.1.

# 3. 절대 금지 10계명 (미륵이의 피로 쓴 것)

1. pending 등록 없이 주문을 전송하지 않는다 — 주문은 OrderGateway 하나로만
2. 시뮬레이터(replay) 검증 없이 배포하지 않는다 — "라이브 미검증"을 기본 상태로 만들지 않는다
3. 장중에 모델을 학습시키지 않는다
4. 장중에 코드를 배포하지 않는다
5. 몽키패치로 메서드를 덮어쓰지 않는다
6. 피처셋 불일치를 DEBUG로 침묵시키지 않는다 — 불일치는 추론 거부다
7. 게이트를 섀도 계측 없이 실차단으로 투입하지 않는다
8. 교정 안 된 확률로 임계값을 정하지 않는다
9. 저장된 종목코드를 신뢰하지 않는다 — 기동마다 프로브, 주문 전 일치 확인
10. 커밋 안 된 수정을 실전 PC에 남기지 않는다 — 배포는 릴리스 번들로만

# 4. 남기는 말

미륵이의 660개 로그는 실패의 기록이 아니라 **취득 원가가 명확한 자산**이다.
같은 수업료를 두 번 내지 않는 것 — 그것이 메시아가 미륵이에게 진 빚을 갚는 방식이다.

> 1차 원정대의 지도를 들고 가는 자는, 늪의 위치를 몰라서 빠지는 일은 없어야 한다.
